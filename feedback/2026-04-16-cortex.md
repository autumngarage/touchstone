# Touchstone Feedback

**From:** cortex dogfood (Phase A, pre-code)
**Touchstone version:** 1.0.1 (brew)
**Date:** 2026-04-16
**Context:** Fresh repo, `touchstone init` had been run once at bootstrap. Walked through what a new contributor would hit on first clone.

---

## Headline ask: one-command install

Right now, going from "fresh clone of a touchstone-managed repo" to "fully gated, review-enabled working state" takes multiple commands, and the gaps aren't surfaced anywhere. Proposal: **`touchstone init` should end with a working, gated repo — no follow-up steps required, or if they're required, printed as a checklist before exit.**

Concretely, a first-run should be one of:

```bash
touchstone init              # existing repo — install everything, report what's missing
touchstone new <dir>         # new project — same, plus scaffold
```

…and when it finishes, the repo should be in one of two states:

1. **Fully armed.** Hooks installed, linters/formatters present, review tooling wired, `touchstone run validate` passes. Print a green summary.
2. **Partially armed with a visible gap list.** Print exactly what's missing and the one command that fixes each gap. No silent under-configuration.

Today we're in a third state: files are present, hooks are not installed, and nothing says so.

---

## Specific friction observed

### 1. `pre-commit install` is not run by `touchstone init`

**Observed.** After `touchstone init`, `.pre-commit-config.yaml` ships with both pre-commit and pre-push stages (including Codex review on default-branch pushes). But `.git/hooks/` is empty:

```
$ ls .git/hooks/ | grep -v sample
(no output)
```

Nothing gates commits or pushes until someone manually runs:

```
pre-commit install --hook-type pre-commit --hook-type pre-push
```

**Why it's bad.** A new clone of a "touchstone-managed" repo is silently ungated. The project looks protected (config files are present, `.codex-review.toml` is there) but in practice anyone can push straight to main with no hook running. This is the most dangerous kind of failure — it looks safe.

**Fix.** `touchstone init` should run `pre-commit install --hook-type pre-commit --hook-type pre-push` as its final step. On existing clones, `touchstone doctor` should detect missing hooks and fail loudly.

### 2. `touchstone run validate` treats "no tests collected" as failure

**Observed.**

```
$ touchstone run validate
  OK ruff not installed; skipped
  OK pyright/mypy not installed; skipped
==> /opt/homebrew/bin/python3 -m pytest
…
collected 0 items
============================ no tests ran in 0.09s =============================
Exit code 5
```

Missing linters skip cleanly with `OK ... not installed; skipped`. But pytest returning exit 5 (no tests collected) bubbles up as a validate failure.

**Why it's bad.** Pre-code phases (Phase A in this repo, or any newly-scaffolded project) have no tests yet. `validate` failing in that state means either (a) `validate` is unusable until someone writes a test, or (b) developers learn to ignore validate output. Both are worse than treating "no tests" as a warning.

**Fix.** Treat pytest exit 5 the same as absent linters — `OK pytest found no tests; skipped` and continue. If a repo wants to enforce "tests must exist," that's a separate policy (`.touchstone-config` flag), not the default.

### 3. `hooks/` directory referenced in docs but not created

**Observed.** CLAUDE.md (touchstone-authored section on Codex merge review) reads:

> If the project has Codex review configured (see `hooks/codex-review.sh` and `.codex-review.toml`)…

But `hooks/` doesn't exist in this repo. The actual script is at `scripts/codex-review.sh`.

**Why it's bad.** Stale paths in template documentation erode trust in the rest of the documentation. A new contributor who follows the reference will `ls hooks/` and find nothing.

**Fix.** Update the template CLAUDE.md section to reference `scripts/codex-review.sh`, or rename/symlink to match. Pick one canonical location.

### 4. No "what happens next" at the end of `touchstone init`

**Observed.** `touchstone init` succeeds silently (from the user's point of view — no terminal output describing what's now true about the repo and what still needs doing).

**Why it's bad.** There's no way to know, without reading the docs, whether the repo is fully set up. See issue #1 above — someone can reasonably assume `init` is complete when it isn't.

**Fix.** End `touchstone init` with a summary block:

```
✓ Installed files: .pre-commit-config.yaml, .codex-review.toml, principles/, scripts/
✓ Installed git hooks: pre-commit, pre-push
✓ Registered project in ~/.touchstone/registry

Next steps:
  • Optional: install ruff (`pip install ruff`) to enable lint checks
  • Optional: install codex CLI to enable pre-push review
  • Run `touchstone run validate` to verify
```

Or when something fails:

```
✗ Could not install pre-push hook: pre-commit not found on PATH
  Fix: `pip install pre-commit && touchstone init --resume`
```

### 5. `touchstone doctor` vs `touchstone update --check` — unclear division

**Observed.** There's a `touchstone doctor` command and a `touchstone update --check` command. From the help text it's not obvious what each diagnoses. In practice, `update --check` answers "am I on the latest touchstone?" and `doctor` answers "is my installation healthy?" — but neither seems to check "are hooks actually installed in this repo?", which is the gap that mattered most here.

**Fix.** Add a third concept — `touchstone status` or `touchstone doctor --project` — that specifically validates per-project state: are hooks installed, are referenced tools on PATH, does the registry entry match reality, are the template files drifted from their source versions.

---

## What worked well

Balancing the above — genuine positives:

- **Graceful skipping for absent optional tools.** `OK ruff not installed; skipped` is exactly the right UX. The inconsistency in issue #2 is that pytest doesn't follow this same pattern, not that the pattern is wrong.
- **`touchstone update --check`** is clean and reassuring: reports "Already up to date" without ceremony. Good diagnostic default.
- **Single-command workflow for shipping** (`scripts/open-pr.sh --auto-merge`) is excellent — push, open PR, review, merge, clean up, sync. This is the one-command experience I want for `init` too.
- **`.touchstone-config` profile detection** pulled in the right Python defaults without asking. Good convention-over-config.

---

## TL;DR for the touchstone team

| Priority | Ask |
|---|---|
| **P0** | `touchstone init` should install git hooks, not just write config files. A "touchstone-managed repo" with no active hooks is a footgun. |
| **P0** | `touchstone run validate` should treat "no tests collected" (pytest exit 5) the same way it treats absent linters: skip gracefully, don't fail. |
| **P1** | End every `touchstone init` with a terminal summary of what's installed and what's still needed — no silent success states. |
| **P1** | Add a project-level health check (`touchstone doctor --project` or similar) that validates hooks-installed, tools-on-PATH, and template-not-drifted. |
| **P2** | Fix the `hooks/` vs `scripts/` reference in the template CLAUDE.md Codex review section. |

The through-line: **reduce the distance between "ran the command" and "repo is actually protected" to zero**. Today there's a gap, and it's invisible.
