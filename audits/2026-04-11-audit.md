## Principles drift

### `engineering-principles.md`

- **Classification**: still-correct, gap-identified
- **Evidence**: Every named rule (no band-aids, no silent failures, derive don't persist, one code path, think in invariants, every fix gets a test, current decisions from current data, audit one weak-point class) is directly endorsed by current Anthropic guidance on agent design and tool authoring. Keep the existing file focused on code correctness; the new file covers agent-facing concerns. See the "New capabilities worth considering" section for a suggested outline.
- **Sources**: [Effective context engineering for AI agents](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents), [Skill authoring best practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices), [Writing effective tools for AI agents](https://www.anthropic.com/engineering/writing-tools-for-agents)

## Implementation drift

### 1. `CHANGELOG.md` is stale — violates `documentation-ownership.md`

- **What**: The CHANGELOG has entries through `0.2.0` (2026-04-09), but `VERSION` is `0.5.12` and `git log` shows explicit release commits for `0.3.0`, `0.4.0`, `0.5.0` through `0.5.12`.
- **Principle touched**: `documentation-ownership.md` — "each document has a clear scope" and "one canonical owner per volatile fact." The CHANGELOG nominally owns release history, but in practice git log has become the canonical source, and the CHANGELOG is a stale second copy.
- **Evidence**: `CHANGELOG.md` lines 7-25 (last entry `0.2.0`); `VERSION` file (`0.5.12`); `git log --oneline -40` shows version tags `v0.5.12, v0.5.11, v0.5.10, v0.5.9, v0.5.8, v0.5.7, v0.5.6, v0.5.5, v0.5.4, v0.5.3, v0.5.1, v0.5.0, v0.4.0, v0.3.0, v0.2.1, v0.2.0`.
- **Classification**: adopt
- **Blast radius**: cosmetic/trust only. Does not affect propagation or release gates. But it's a live violation of a principle the toolkit preaches to downstream projects, which weakens the principle's credibility.
- **Recommendation**: either (a) backfill the CHANGELOG from git log through `0.5.12`, or (b) explicitly retire the CHANGELOG and declare git log + GitHub Releases as the canonical owner of release history in `CLAUDE.md` and `README.md`. Option (b) is more aligned with the principle ("when in doubt, delete the duplicate") but option (a) preserves the human-readable narrative. Pick one deliberately; the current state picks neither.

### 2. `scripts/codex-review.sh` is a 9-line wrapper — NOT a violation (noted positively)

- **What**: Spot-check found two files named `codex-review.sh` — one in `hooks/`, one in `scripts/`. On initial read this looked like a "one code path" violation.
- **Evidence**: `scripts/codex-review.sh:1-9` is a thin wrapper that execs `$SCRIPT_DIR/../hooks/codex-review.sh`. The canonical implementation (718 lines) lives only in `hooks/`. `bootstrap/new-project.sh:231` copies FROM `hooks/codex-review.sh` TO downstream projects' `scripts/codex-review.sh`.
- **Classification**: already-have
- **Note**: this is a good example of the spot-check methodology — the violation was apparent at directory-listing level and cleared at code-reading level. Including it here as a positive note.

### 3. `chmod +x "$PROJECT_DIR/setup.sh" 2>/dev/null || true` — minor silent failure

- **What**: `bootstrap/new-project.sh:219` suppresses both stderr and exit code on `chmod +x`.
- **Principle touched**: `engineering-principles.md` — "no silent failures."
- **Evidence**: single line at `bootstrap/new-project.sh:219`.
- **Classification**: needs-discussion
- **Blast radius**: the file is a high-scrutiny path (AGENTS.md declares `bootstrap/new-project.sh` as a high-scrutiny file). A chmod failure on macOS/Linux would silently leave `setup.sh` non-executable, and the user's next step (`bash setup.sh`) would still work because it invokes through `bash`, not via exec. So the failure is *tolerable* but still masked.
- **Recommendation**: consider whether cross-platform (NTFS/FAT) fallback is actually needed. If not, remove the suppression so a real failure surfaces. If yes, log the failure to stderr at INFO level rather than silence it entirely. Either way, annotate the line with the reason (the principle says silent-failure exceptions should be deliberate).

### 4. `lib/auto-update.sh` version comparison is string equality — brittle edge case

- **What**: `lib/auto-update.sh:62` uses `[ "$current_version" = "$latest_version" ]`. Comment acknowledges the limitation: "simple string compare — works for semver if we don't skip versions."
- **Principle touched**: `engineering-principles.md` — "think in invariants." The invariant here is "current ≥ latest should mean no upgrade," but string inequality conflates "current > latest" with "current < latest."
- **Evidence**: `lib/auto-update.sh:61-79`. When running from a git clone ahead of the latest release (e.g., developing on `0.5.13-pre` while main ships `0.5.12`), string inequality triggers the upgrade path. In practice `brew upgrade` is a no-op on a non-brew install, and `git pull --rebase` is a no-op on already-pulled commits — so this works today. But it's a fragile coupling between the version-compare semantics and the upgrade mechanism's tolerance for no-ops.
- **Classification**: needs-discussion
- **Blast radius**: low (runs per-command, fail-open on any error). But the principle cost is non-zero because it normalizes fragile version handling for downstream projects that might copy the pattern.
- **Recommendation**: consider a semver-aware compare (sort by `version_sort` or equivalent) and short-circuit if `current >= latest`. Low priority.

### 5. Fail-open patterns in Codex hook and auto-update — still-correct

- **What**: Both `hooks/codex-review.sh` and `lib/auto-update.sh` silently skip on infrastructure failures (network down, tool missing, parse error).
- **Principle touched**: `engineering-principles.md` — "no silent failures" on its face conflicts with fail-open. But `AGENTS.md:11,35-36` explicitly declares fail-open as the contract for the Codex hook ("The hook must never block a push due to its own infrastructure failure — only on actual code review findings").
- **Classification**: already-have (deliberate, documented)
- **Note**: this is worth explicitly marking as a *deliberate exception* in `engineering-principles.md` itself, not just in `AGENTS.md`. Currently the principle is absolute in `principles/` but the exception lives in `AGENTS.md`, creating a small disconnect between what the toolkit teaches and what the toolkit practices. Consider adding one sentence to the "No silent failures" section: *"Exception: infrastructure gates (review hooks, auto-update checks) may fail open — but only when the exception is declared in the project's AGENTS.md with the justification."*

### 6. Test coverage across `lib/`, `bootstrap/sync-all.sh`, several `scripts/` — needs-investigation

- **What**: Named self-tests exist for `bootstrap/new-project.sh`, `bootstrap/update-project.sh`, `scripts/merge-pr.sh`, `hooks/codex-review.sh`, and ADR templates (`test-adr.sh`). No file-level test exists for `lib/auto-update.sh`, `lib/release.sh`, `lib/colors.sh`, `bootstrap/sync-all.sh`, `scripts/cleanup-branches.sh`, `scripts/open-pr.sh`, `scripts/run-pytest-in-venv.sh`, or `bin/toolkit`.
- **Principle touched**: `engineering-principles.md` — "every fix gets a test." Note: the principle is about *fixes*, not about every file having a test. Whether this is drift depends on whether historical fixes to these files added tests elsewhere.
- **Classification**: needs-investigation
- **Blast radius**: the untested files include high-impact surfaces (release automation, auto-update, CLI entrypoint). If they have been edited without tests being added, that's drift from the principle.
- **Recommendation**: a follow-up audit pass could grep commit history for changes to these files and check whether each commit also touched a `tests/*.sh` file. That would be a concrete "audit-one-weak-point-class" exercise.

## Process & workflow

### Workflow 1: Git lifecycle

- **Current shape**: feature branch → `scripts/open-pr.sh` (push + `gh pr create`) → `scripts/merge-pr.sh <N>` (mergeability check → Codex review → `--match-head-commit` squash merge → local sync) → `scripts/cleanup-branches.sh`.
- **Classification**: keep-as-is
- **Evidence**: `scripts/merge-pr.sh:163` uses `--match-head-commit "$REVIEWED_HEAD_OID"` to guard against TOCTOU between review and merge. This is a non-obvious correctness property that matches current Anthropic thinking on "solve, don't punt." The mergeability retry loop (`scripts/merge-pr.sh:130-146`) is well-formed. The workflow is sound.
- **Recommendation**: none. This workflow is at a local maximum for a human+AI single-pair-per-branch model. Any refinement would need to be driven by a shift in workflow assumptions (e.g., parallel sub-agents — see Workflow 2).

### Workflow 2: Codex review hook (merge gate + default-branch pre-push)

- **Current shape**: `hooks/codex-review.sh` (718 lines) runs `codex exec --full-auto` against the diff, handles CLEAN/FIXED/BLOCKED sentinels, auto-fixes safe paths, blocks unsafe paths, loops up to `max_iterations` with caching. Recent work (git log) focused on refinement: merge-gate integration (#6), caching clean reviews (#5), preventing nested runs (#7), branch-block scoping.
- **Classification**: needs-discussion
- **Evidence**: The Codex hook is a serial, single-reviewer gate. Anthropic's [sub-agents guide](https://docs.claude.com/en/docs/claude-code/sub-agents) and [Claude Code advanced patterns webinar](https://www.anthropic.com/webinars/claude-code-advanced-patterns) explicitly recommend *parallel* sub-agents for code review (style-checker, security-scanner, test-coverage) with isolated contexts. The toolkit's current pattern cannot do this because Codex is a single external CLI invocation.
- **Recommendation**: consider a **complementary** (not replacement) Claude-sub-agent review gate. Two concrete options to weigh:
  - **Option A (low risk)**: add a new hook `hooks/claude-review.sh` that runs a small set of Claude sub-agents in parallel (e.g., `security-review`, `principles-check`, `test-coverage`) *before or alongside* the Codex run. Keep Codex for final arbitration. This preserves the existing Codex gate investment while gaining parallelism.
  - **Option B (higher risk)**: migrate review entirely to Claude sub-agents, retire Codex as a merge-gate dependency. Faster but loses the cross-model second-opinion property that having Codex currently provides.
  - Option A is the more toolkit-native answer ("one code path" and "don't rip out working infra"). Option B would be larger blast radius.
- **Gap in this evaluation**: Codex CLI is OpenAI, not Anthropic, and was therefore excluded from the research phase per the skill's hard constraint. A full Codex-best-practices check would require loosening that restriction.

### Workflow 3: Release flow (`lib/release.sh` + `bin/toolkit release` + Homebrew tap)

- **Current shape**: `toolkit release --patch|--minor|--major` bumps `VERSION`, commits, pushes, creates GitHub Release, updates Homebrew formula. CLAUDE.md declares a four-way agreement invariant (GitHub Releases, Homebrew formula, `origin/main`, local brew install must all agree on vX.Y.Z before a release is "done").
- **Classification**: keep-as-is
- **Evidence**: The invariant is unusually explicit and well-formed for a solo project. It catches the most common release bug (partial rollout). The recent commit "Release script: clone tap to temp dir instead of requiring local checkout" shows the script has been hardened for portability.
- **Recommendation**: none functional. Documentation-side: the release invariant should also be cross-referenced in CHANGELOG (once CHANGELOG is fixed per implementation-drift finding #1).

### Workflow 4: Bootstrap & propagation (`bootstrap/new-project.sh`, `update-project.sh`, `sync-all.sh`, `~/.toolkit-projects`)

- **Current shape**: A project is bootstrapped once via `toolkit new <dir>`, registering in `~/.toolkit-projects`. Updates propagate via `toolkit sync` (or `bootstrap/sync-all.sh`) which iterates the registry and runs `update-project.sh` in each. Toolkit-owned files (`principles/*`, `scripts/*`, `hooks/codex-review.sh`) are copy-overwritten with `.bak` backups; project-owned files (`CLAUDE.md`, `AGENTS.md`, `.codex-review.toml`) are never touched after bootstrap.
- **Classification**: **needs-discussion** — this is the single highest-impact finding in the audit.
- **Evidence**: Claude Code now ships a [plugin marketplace](https://code.claude.com/docs/en/plugin-marketplaces) in public beta (as of 2026). A plugin marketplace is a git repo with `.claude-plugin/marketplace.json` that distributes skills, agents, hooks, MCP servers, and LSP servers — exactly the primitives the toolkit currently file-copies. Users install with `/plugin marketplace add henrymodisett/toolkit`. The plugin system provides native version tracking, automatic updates, and centralized discovery — three things the toolkit currently reinvents.
- **Recommendation**: **this is a strategic question, not a refinement.** Frame as three options for user decision:
  - **Option 1 — Stay pure file-copy**: the current approach. Pros: works outside Claude Code (the `setup.sh` + git hooks work in any shell); portable to projects that don't use Claude Code; no dependency on Claude Code being installed. Cons: reinvents update discovery; can't cleanly distribute skills/agents/MCP servers; users without the toolkit CLI can't benefit.
  - **Option 2 — Dual distribution**: keep file-copy for scripts and principles (which need to work in bare shells), but *also* publish a companion plugin marketplace at `.claude-plugin/marketplace.json` in the toolkit repo that exposes the Claude-Code-native surface (the `toolkit-audit` skill, hypothetical future skills, maybe the principles as `.claude/rules/` imports). Users who want Claude Code native get the plugin; users who want bare-shell tooling get the existing file-copy path. Higher maintenance burden (two distribution paths) but preserves the existing investment.
  - **Option 3 — Pivot to plugin-first**: rewrite the toolkit as a plugin marketplace where scripts become skills, principles become `.claude/rules/` files, hooks become Claude Code hooks. Retire `~/.toolkit-projects`, `sync-all.sh`, file-copy propagation. Highest blast radius; largest payoff; loses bare-shell use cases.
  - Option 2 is probably the right starting point because it's additive rather than destructive. The toolkit already dogfoods — the new `toolkit-audit` skill is a natural first member of a marketplace. But this is a judgment call that should be made deliberately, not incrementally.
- **Blast radius**: any choice here changes how every downstream project consumes the toolkit. This is a top-level strategic decision and should not be made in a single PR — it warrants a short design doc or ADR first.

### Workflow 5: Auto-update (`lib/auto-update.sh`, `~/.toolkit/last-update-check`)

- **Current shape**: On every `toolkit` invocation, check GitHub Releases API for newer version. Throttled to 1x/hour via file-based timestamp cache. Fail-open on any error (network, parse, upgrade).
- **Classification**: keep-as-is
- **Evidence**: The design is sound: file-based throttle cache is a justified exception to "derive don't persist" (GitHub API rate limits). Fail-open matches the toolkit's declared contract for infrastructure gates. See implementation-drift finding #4 for a minor nit on the version-compare semantics.
- **Recommendation**: none structural. If you adopt Option 2 or 3 from Workflow 4 (plugin marketplace), auto-update becomes partially redundant because Claude Code plugins have native update discovery. But don't retire `lib/auto-update.sh` until that migration is actually chosen.

### Workflow 6: Testing discipline (`tests/test-*.sh`)

- **Current shape**: Shell-based self-tests for bootstrap, update, merge-pr, review hook, ADR templates. `CLAUDE.md` mandates running all `tests/test-*.sh` before any push.
- **Classification**: keep-as-is
- **Evidence**: shell tests are portable, fast, and match the shell-script nature of the toolkit. The mandate to run all tests before push is strong discipline.
- **Recommendation**: if the toolkit gains more skills (per Workflow 4 Option 2 or 3), consider adding the [skill evaluation pattern](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices) — 3+ test scenarios per skill with expected behavior, run against the skill with a real Claude instance — rather than stretching shell tests to cover skill behavior. Skills need a different test shape (behavioral, not functional).

## New capabilities worth considering

### 1. New principle file: `principles/context-engineering.md`

- **What**: A new principle alongside `engineering-principles.md` specifically covering AI-agent-facing concerns. The existing principles are 100% about code correctness; 2026-era agent work has a parallel discipline around context curation that isn't represented.
- **Suggested outline** (3-5 rules, keep crisp in the toolkit's house style):
  - **Right altitude for system prompts**: neither brittle if-else nor too-vague. Start minimal, add clarity based on observed failure modes.
  - **Minimal viable tool set**: a bloated tool set is an anti-pattern; if a human can't decide which tool to use, neither can an agent.
  - **Just-in-time retrieval**: prefer lightweight identifiers + dynamic lookup over eager pre-fetching. Context is finite.
  - **Context hygiene for long tasks**: compaction, sub-agents, or structured notes — not "hope the window holds."
  - **No prompt injection from tool results**: treat tool output as untrusted input. Flag suspected injection before acting.
- **Where it fits**: `principles/context-engineering.md`, referenced from `CLAUDE.md` under a new "## Context Engineering" section, and imported into downstream projects via the standard propagation mechanism.
- **Blast radius**: propagates to every downstream project. High-impact but purely additive (no existing principle is removed).
- **Sources**: [Effective context engineering for AI agents](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents), [Writing effective tools for AI agents](https://www.anthropic.com/engineering/writing-tools-for-agents)

### 2. Plugin marketplace distribution (companion, not replacement)

- **What**: Publish a `.claude-plugin/marketplace.json` in the toolkit repo root so the toolkit can be installed into any Claude Code session via `/plugin marketplace add henrymodisett/toolkit`. Initial marketplace members: the new `toolkit-audit` skill (already built), and any future skills the toolkit produces.
- **Where it fits**: `.claude-plugin/marketplace.json` at the toolkit repo root. Does not require moving existing files.
- **What it replaces/augments**: complements `sync-all.sh` for Claude-Code-native users; does not replace the file-copy path for bare-shell users.
- **Why now**: you already built one skill (`toolkit-audit`) and will likely build more. A marketplace is the native home for skills to live in.
- **Blast radius**: low if treated as additive (Option 2 from Workflow 4). High if treated as a pivot (Option 3).
- **Sources**: [Create and distribute a plugin marketplace](https://code.claude.com/docs/en/plugin-marketplaces), [Customize Claude Code with plugins](https://www.anthropic.com/news/claude-code-plugins)

### 3. Parallel Claude sub-agents as a review gate (augmenting Codex)

- **What**: A small set of Claude sub-agents running in parallel on a PR diff (`security-review`, `principles-check`, `test-coverage`), each with its own isolated context and a focused system prompt. Results synthesized into a single verdict that feeds into — not replaces — the existing Codex gate.
- **Where it fits**: new script `hooks/claude-review.sh` alongside `hooks/codex-review.sh`; invoked from `scripts/merge-pr.sh` before the Codex run.
- **Why it's interesting**: Anthropic explicitly recommends this shape for code review. The toolkit's review gate is currently serial and cross-model diversity comes only from Codex; adding a parallel Claude-side check adds both speed and a complementary perspective.
- **Blast radius**: moderate — adds a new code path in the review gate, but behind a config flag so it can be opt-in per project.
- **Sources**: [Create custom subagents](https://docs.claude.com/en/docs/claude-code/sub-agents), [Subagents in the SDK](https://docs.claude.com/en/docs/agent-sdk/subagents)

### 4. `.claude/rules/` split for modular instructions

- **What**: Claude Code's [memory documentation](https://code.claude.com/docs/en/memory) describes a `.claude/rules/` convention for splitting project instructions into multiple files rather than one monolithic CLAUDE.md. The toolkit's current CLAUDE.md template is 66 lines (well under the 200-line target), so no immediate action. But downstream projects using the toolkit may accumulate large CLAUDE.md files — the template could include a commented-out `@.claude/rules/*.md` import pattern as a forward hint.
- **Classification**: add-capability, low priority
- **Sources**: [How Claude remembers your project](https://code.claude.com/docs/en/memory), [Best Practices for Claude Code](https://code.claude.com/docs/en/best-practices)

### 5. MCP server for the toolkit registry

- **What**: Expose `~/.toolkit-projects`, project staleness, and per-project health as an MCP server. Any Claude Code session could then query the toolkit's state without running shell commands.
- **Classification**: needs-discussion, low priority
- **Rationale**: speculative. The current `toolkit doctor` and `toolkit status` commands already cover this; an MCP server would only be a win if it enabled cross-session workflows that aren't possible today.

### 6. Refinement of the `toolkit-audit` skill itself

- **What**: The skill I just wrote (today, 2026-04-11) has two minor deviations from current skill authoring best practices:
  - **Description is imperative, not strict third person**. Current: `"Audit the toolkit project..."`. Preferred per [skill authoring best practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices): `"Audits the toolkit project..."`. Minor.
  - **No evaluations created before writing the skill**. The recommended workflow is "Create evaluations BEFORE writing extensive documentation." This skill was written without a baseline eval.
  - **No progressive disclosure split**. SKILL.md is ~180 lines (under the 500-line limit so not a hard problem) but the phase details, research topic list, and report template could be split into `phases/phase-2-research.md`, `phases/phase-3-evaluation.md`, and `templates/report-template.md` for easier iteration.
- **Classification**: needs-discussion
- **Recommendation**: before investing in refinement, observe the skill in real use (this run being the first). If the current structure works, keep it; if Claude struggles with any phase, that's the signal to split it out per the progressive-disclosure pattern.

## Gaps in this audit

1. **Codex CLI best practices were not researched.** The skill's hard constraint restricts research to `docs.claude.com`, `docs.anthropic.com`, and `anthropic.com`. Codex is OpenAI-owned and therefore out of scope. A comprehensive Codex review of `hooks/codex-review.sh` would require loosening that restriction or a separate audit pass.
2. **Test coverage for several `lib/`, `scripts/`, `bootstrap/sync-all.sh`, and `bin/toolkit` files was not exhaustively checked.** Listed as `needs-investigation` in implementation-drift finding #6. A follow-up audit could grep commit history for changes to these files and verify whether each change added a corresponding test.
3. **The spot-check of implementation drift was shallow by design.** The skill explicitly says "do not exhaustively audit." A full audit pass against `hooks/codex-review.sh` (718 lines) for error-swallowing, band-aids, or silent-failure patterns could produce additional findings.
4. **Downstream projects were not audited.** The skill's scope is the toolkit repo only. Projects registered in `~/.toolkit-projects` may have drifted from the toolkit in ways a toolkit-only audit cannot see.
5. **The new `toolkit-audit` skill was not tested across models.** Only executed once, with Opus 4.6. Whether it works with Haiku 4.5 or Sonnet 4.6 is unknown.
